# Phase 4 — UI Design Contract

> Visual and interaction contract for the battery picker, comparison table, saldering
> columns, coarse-cadence banner, period-narrowing control, and "Rekenen..." indicator.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Scope note:** Phase 4 ships the first real interactive UI on top of the proven Phase 3
> engine. It re-wires `#drop-zone-region` and fills `#results-region` using `@preact/signals-core`.
> uPlot charts are Phase 5 — do NOT add them here.

---

## Design System

| Property | Value | Source |
|----------|-------|--------|
| Tool | none | CLAUDE.md — vanilla TS + Vite, no component library |
| Preset | not applicable | — |
| Component library | none | CLAUDE.md constraint |
| Icon library | none — Unicode glyphs only (▸ for expand, ✕ for close) | default — system-font stack satisfies CSP |
| Font | System-font stack (inherited from `--font-sans` token) | tokens.css (locked Phase 1) |

**Implementation:** All new styles go into new CSS files imported by `src/styles/global.css`
or directly by the module that owns them. Zero inline `style=` attributes anywhere —
`style-src 'self'` CSP is a hard constraint. All state changes via CSS class swaps.

## Spacing Scale

Inherited from Phase 1 tokens.css — no new tokens needed. All Phase 4 layout uses these
existing CSS custom properties:

| Token | CSS Custom Property | Value | Phase 4 Usage |
|-------|--------------------|----|---------------|
| xs | `--space-xs` | 4px | Icon gaps, swatch–label gap, cell inner padding (compact) |
| sm | `--space-sm` | 8px | Between spec rows inside a battery card, between table cells |
| md | `--space-md` | 16px | Card inner padding, between picker cards, date-range label–input gap |
| lg | `--space-lg` | 24px | Section gap between picker and table, between table and footer notes |
| xl | `--space-xl` | 32px | Gap between major regions (picker block, table block, period block) |
| 2xl | `--space-2xl` | 48px | Inherited shell region margins (unchanged) |
| 3xl | `--space-3xl` | 64px | Not used in Phase 4 components |

Exceptions:
- Touch target minimum: 44px height on all interactive elements (WCAG 2.5.5). Spec cards
  with checkboxes, the date-range inputs, and the "± eigen batterij" expand trigger must
  all meet this minimum. Applied as a one-off rule, not a new token.
- Comparison table cells: `padding: var(--space-xs) var(--space-sm)` (4px × 8px) for
  compactness when 8 columns are visible. This is a table-specific override, not a new token.

## Typography

Inherited from Phase 1 tokens.css. No new sizes or weights declared.

| Role | CSS Custom Property | Size | Weight | Line Height | Phase 4 Usage |
|------|--------------------|----|--------|-------------|---------------|
| Body | `--font-size-body` | 16px | 400 (regular) | 1.5 | Card spec values, table cell values, date-range inputs, status text |
| Label | `--font-size-label` | 14px | 400 (regular) | 1.4 | Column headers, spec keys inside cards, "X dagen aan data" coverage text, disclaimer copy |
| Heading | `--font-size-heading` | 20px | 600 (semibold) | 1.2 | Section headings ("Kies batterijen", "Resultaten") |
| Display | — | — | — | — | Not used in Phase 4 |

**Weights declared: 2** — 400 (regular) and 600 (semibold). No new weights.
**Sizes declared: 3** — 14px, 16px, 20px. This is the Phase 1 contract; Phase 4 does not extend it.

**Number formatting rule (spirit of VIZ-04, applied now):** All computed kWh values
rendered with exactly 1 decimal place (`toFixed(1)`) and a `kWh` suffix. Percentages
at 1 decimal place with a `%` suffix. No raw floats like "1.7432 kWh" anywhere.

Leader-highlight boldness: cells that are the per-column leader use `font-weight: var(--font-weight-semibold)` (600). This is the ONLY place semibold appears inside the
table body — all other body cells use regular (400). This contrast is sufficient to
call out the leader without color.

## Color

All values are existing CSS custom properties from `tokens.css`. No new color tokens.

| Role | CSS Custom Property | Value | Phase 4 Usage |
|------|--------------------|----|---------------|
| Dominant (60%) | `--color-bg` | `#ffffff` | Page background, card background (default), table background |
| Secondary (30%) | `--color-surface` | `#f4f4f5` | Picker section background, leader cell tint, disabled card background |
| Border | `--color-border` | `#e4e4e7` | Card borders, table row/column separators, section dividers |
| Text primary | `--color-text` | `#18181b` | Card battery name, table cell values, section headings |
| Text muted | `--color-text-muted` | `#71717a` | Spec keys inside cards, column header labels, coverage text, disclaimer text, "Rekenen..." text |
| Accent (10%) | `--color-accent` | `#2563eb` | Focus rings on interactive elements only; NOT used for headings or swatches |
| Destructive | `--color-destructive` | `#dc2626` | Coarse-cadence banner background tint + left border; negative saldering-ON values |

Accent (`#2563eb`) reserved for: keyboard focus rings on checkboxes, the expand
trigger, date inputs, and any future link. NOT used for battery swatches, banner
backgrounds, section headings, or table leader highlights.

Destructive (`#dc2626`) reserved for: the coarse-cadence warning banner (left border +
background tint at 8% opacity) and negative saldering-ON "kWh netto-import vermeden"
cell values where the value is ≤ 0 (to signal "no benefit"). The value is shown as-is
with the destructive text color — it is NOT floored at 0.

### Per-Battery Color Slots (COMP-04)

Declared in Phase 1 tokens.css. Phase 4 reads them via the `colorFor(batteryId)` helper
(tested in Vitest node env). The mapping is deterministic and stable into Phase 5.

| Slot | CSS Custom Property | Hex Value | Assigned Battery (by catalog order) |
|------|--------------------|----|--------------------------------------|
| 1 | `--color-battery-1` | `#2563eb` | Sessy 5 kWh (default / index 0) |
| 2 | `--color-battery-2` | `#16a34a` | Sessy 10 kWh |
| 3 | `--color-battery-3` | `#d97706` | Zonneplan 10 kWh |
| 4 | `--color-battery-4` | `#9333ea` | Tesla Powerwall 3 |
| 5 | `--color-battery-5` | `#e11d48` | Huawei LUNA / Victron / Marstek / Custom |

**colorFor() mapping rule:** Slot is assigned by the position of the battery in the
user's CURRENT SELECTION (1-indexed, order of selection). Catalog position does not
determine slot — selection order does. This means:
- User selects Sessy 5 first → slot 1 (#2563eb).
- User deselects Sessy 5 then reselects → slot 1 is re-assigned to the next selected
  battery until Sessy 5 is re-added (in its new position).
- The custom battery always takes the next available slot in selection order.

**Color-blind safety:** The five hues (blue, green, amber, purple, rose) are chosen
to remain distinguishable for deuteranopia (red-green) and protanopia color-blindness.
Blue (#2563eb) and green (#16a34a) differ in luminance; amber (#d97706) and rose (#e11d48)
differ in hue; purple (#9333ea) is perceptually isolated. Verified against the
WCAG 3:1 non-text contrast requirement on `--color-surface` backgrounds.

**Swatch rendering:** Each battery card and table row header shows a 12×12px filled
circle with `border-radius: 50%` using the assigned battery color as `background-color`
set via a CSS class `.battery-swatch--1` through `.battery-swatch--5`. NOT via inline
`style=` (CSP). Each `.battery-swatch--N` class sets `background-color` to the
corresponding `--color-battery-N` token.

**Leader highlight treatment:** Leader cells (best value per metric column) receive
class `.table-cell--leader` which applies:
- `font-weight: var(--font-weight-semibold)` (600)
- `background-color: var(--color-surface)` (neutral tint — the same surface color used elsewhere)
- No border change, no color-battery-N color — keeps leader separate from identity (D-11)

## Component Inventory

### 1. Battery Picker Section

**Container:** `<section aria-label="Batterijkeuze">` rendered inside `#drop-zone-region`
(Phase 4 re-wires this region — D-16). The `p.privacy-promise` child is preserved verbatim.

**Section heading:** `<h2>` "Kies batterijen" at `--font-size-heading` (20px), semibold.

**Card grid:** `<ul role="list" class="battery-picker">` — CSS grid, 1 column on narrow,
2 columns on ≥ 480px viewports. Each catalog battery is one `<li class="battery-card">`.

**Spec card anatomy** (D-05):
```
<li class="battery-card [battery-card--selected] [battery-card--disabled]">
  <label class="battery-card__label">
    <input type="checkbox" class="battery-card__checkbox" ...>
    <span class="battery-card__swatch battery-swatch--N"></span>
    <span class="battery-card__name">Sessy 5 kWh</span>
  </label>
  <dl class="battery-card__specs">
    <dt>Capaciteit</dt>  <dd>5,0 kWh</dd>
    <dt>Bruikbaar</dt>   <dd>100%</dd>
    <dt>Rendement</dt>   <dd>85%</dd>
    <dt>Max laden</dt>   <dd>2,2 kW</dd>
    <dt>Max ontladen</dt><dd>1,7 kW</dd>
  </dl>
</li>
```

Spec keys use `--font-size-label` (14px), muted color. Spec values use `--font-size-body`
(16px), primary text color.

**Card states:**
- Default: `--color-bg` background, 1px solid `--color-border` border, border-radius 8px.
- Selected (checkbox checked): 2px solid `--color-accent` border, `--color-bg` background.
  The accent border is the ONLY place accent appears on cards.
- Disabled (5 already selected, this card is not among them): `--color-surface` background,
  `--color-text-muted` text, `cursor: not-allowed`, `opacity: 0.6`.
- Sessy 5 kWh is pre-checked on mount (BATT-03).

**Max-5 cap note** (BATT-05): When 5 batteries are selected, a `<p class="picker-cap-note">`
appears below the grid: "Maximaal 5 batterijen geselecteerd." at `--font-size-label` (14px),
`--color-text-muted`. Disabled cards show `title="Deselecteer een batterij om een andere te kiezen"`.

### 2. Custom Battery Card (D-06)

**Position:** Last `<li>` in the battery-picker grid, always rendered.

**Collapsed state:** Shows a label "+ Eigen batterij" with a disclosure button
(`<button aria-expanded="false" class="battery-card__expand">`) at min-height 44px.
Counts as one of the 5 when expanded AND the user has filled at least the capacity field.

**Expanded state (aria-expanded="true"):** Shows the 5 input fields inline within the
card body, matching the spec-card visual layout:

```
<form class="custom-battery-form" novalidate>
  <label>Capaciteit (kWh)  <input type="number" min="0.1" step="0.1" placeholder="5,0"></label>
  <label>Bruikbaar (%)     <input type="number" min="1"   step="1"   placeholder="100"></label>
  <label>Rendement (%)     <input type="number" min="1"   step="1"   placeholder="85"></label>
  <label>Max laden (kW)    <input type="number" min="0.1" step="0.1" placeholder="2,2"></label>
  <label>Max ontladen (kW) <input type="number" min="0.1" step="0.1" placeholder="1,7"></label>
</form>
```

**Inline validation:** Invalid fields get class `.input--invalid` which applies
`border-color: var(--color-destructive)`. An associated `<span role="alert" class="input-error">`
below the field shows the Dutch validation message at `--font-size-label`, `--color-destructive`.
Validation fires on `blur` + on any recompute attempt. Do NOT validate on every keystroke
(the debounced recompute handles mid-typing gracefully).

**Placeholder defaults:** Sessy 5 kWh specs as placeholder text so the user sees
realistic defaults even before typing.

**Custom battery is included in the comparison** once it has a valid capacity value (> 0).
Incomplete / invalid form → custom battery excluded from `runComparison` call; a
`<span role="alert">` below the form: "Eigen batterij niet meegenomen — vul minimaal de capaciteit in."

### 3. Period-Narrowing Control (DATA-12 / D-19)

**Placement:** Above the comparison table, below the battery picker. `<section aria-label="Analyseperiode">` with heading `<h2 class="results-section-heading">` "Analyseperiode" at heading size (20px), semibold.

**Control:** Two `<input type="date">` fields (NL date picker — browser native):
```
<label>Van  <input type="date" class="period-input" id="period-from"></label>
<label>Tot  <input type="date" class="period-input" id="period-to"></label>
```
Both default to the full merged dataset range on initial mount. Both have `min` / `max`
attributes set to the merged series boundaries. HTML5 native date picker is used — no
custom calendar widget.

**Coverage indicator** (COMP-08): A `<p class="period-coverage" aria-live="polite">`
adjacent to the date inputs, updated reactively:
"43 dagen aan data" (singular: "1 dag aan data"). This is the only period framing —
no "/jaar" or "/maand" extrapolation anywhere (COMP-07).

**Live recompute:** Any change to either date field triggers `filterByPeriod()` (pure
function, main thread — it is cheap) followed by a debounced signal update that fires
the Comlink worker. The coverage indicator updates immediately on date change (before
the worker responds).

### 4. Coarse-Cadence Warning Banner (D-13)

**Trigger:** `SimResult.coarseCadenceWarning === true` for ANY of the selected batteries.

**Placement:** Between the period-narrowing control and the comparison table. Always
above the table — never below, never in a footnote.

**Anatomy:**
```html
<div role="alert" class="cadence-banner">
  <p class="cadence-banner__text">
    Dagdata overschat sterk wat een batterij opvangt —
    upload 15-minuten P1-data voor een betrouwbare schatting.
  </p>
</div>
```

**Styling:**
- `border-left: 4px solid var(--color-destructive)`
- `background-color: rgb(220 38 38 / 0.08)` (destructive at 8% opacity — one-off rule)
- Padding: `var(--space-md)`
- Border-radius: 4px (left side rounded by border-left convention — right side 4px)
- Text: `--font-size-body` (16px), `--color-text` (primary — not muted; this is prominent)
- Results below remain visible and usable — not blurred, not hidden. The banner caveats,
  it does not block.

### 5. Comparison Table (COMP-01..08 / D-08..D-11)

**Container:** `<section id="results-region" aria-label="Vergelijkingsresultaten">` —
the Phase 1 shell region, now filled by Phase 4.

**Table structure:** `<table class="comparison-table">` — semantic HTML table.
Batteries = rows. Metrics = columns. (D-08)

**Column set** (left→right, in order — D-10):

| # | HTML `<th>` label | `<th>` sub-label | Source | Saldering |
|---|------------------|-----------------|--------|-----------|
| 1 | Battery | (color swatch + name) | row header | — |
| 2 | kWh netto-import vermeden | zonder saldering | derived | OFF (primary) |
| 3 | kWh netto-import vermeden | met saldering | derived | ON (muted) |
| 4 | Zelfverbruik | % | `shiftedKwh / totalImportKwh × 100` | — |
| 5 | Verschoven | kWh | `shiftedKwh` | — |
| 6 | Rest-import | kWh | `residualImportKwh` | — |
| 7 | Rest-teruglevering | kWh | `residualExportKwh` | — |
| 8 | Marginale benutting | kWh/kWh | `shiftedKwh / usableCapacityKwh` | — |

**Column 2 ("zonder saldering")** is the primary / emphasized column — it reads first, is
visually heavier than column 3. Column 3 ("met saldering") is muted — rendered in
`--color-text-muted` for both its header and its cell values, EXCEPT when the value
is ≤ 0 (then `--color-destructive` text, per D-02).

**Columns 2+3 header group:** A `<colgroup>` containing columns 2 and 3 receives class
`.col-saldering-group`. A `<th colspan="2" class="col-saldering-group__header">` acts as
a parent header label: "kWh netto-import vermeden" spanning both columns. Below it, two
child `<th>` cells: "zonder saldering" (primary) and "met saldering" (muted). The
saldering disclaimer trigger (info button) is placed inside the parent `<th>`.

**Table markup skeleton:**
```html
<table class="comparison-table">
  <thead>
    <tr class="thead-group-row">
      <th rowspan="2">Batterij</th>
      <th colspan="2" class="col-saldering-group__header">
        kWh netto-import vermeden
        <button class="saldering-info-btn" aria-expanded="false"
                aria-controls="saldering-disclaimer">i</button>
      </th>
      <th rowspan="2">Zelfverbruik %</th>
      <th rowspan="2">Verschoven kWh</th>
      <th rowspan="2">Rest-import kWh</th>
      <th rowspan="2">Rest-teruglevering kWh</th>
      <th rowspan="2">Marginale benutting kWh/kWh</th>
    </tr>
    <tr class="thead-label-row">
      <th class="col-primary">zonder saldering</th>
      <th class="col-muted">met saldering</th>
    </tr>
  </thead>
  <tbody>
    <!-- one <tr class="battery-row"> per selected battery -->
  </tbody>
</table>
```

**Per-battery row:**
```html
<tr class="battery-row">
  <td class="battery-row__name">
    <span class="battery-swatch battery-swatch--N"></span>
    <span class="battery-row__label">Sessy 5 kWh</span>
  </td>
  <td class="col-primary [table-cell--leader]">12,3 kWh</td>
  <td class="col-muted [table-cell--negative]">−0,4 kWh</td>
  <td class="[table-cell--leader]">42,1 %</td>
  <td>312,5 kWh</td>
  <td>187,2 kWh</td>
  <td>95,3 kWh</td>
  <td class="[table-cell--leader]">62,5</td>
</tr>
```

**Leader highlight logic** (D-11, COMP-03): Per metric column (columns 2–8 independently),
the cell with the best value for that metric receives class `.table-cell--leader`.
"Best" means highest for columns 2, 3, 4, 5, 8 (more is better); lowest for columns 6, 7
(less residual import/export is better). The class applies semibold weight +
`--color-surface` background tint. This is SEPARATE from battery identity color (D-11).

**Negative saldering-ON cell** (D-02): When column 3 value ≤ 0, add class
`.table-cell--negative` which sets `color: var(--color-destructive)`. The value is
displayed as-is (e.g. "−0,4 kWh" with a proper minus sign U+2212). The table row note
(see copywriting) appears BELOW the table — not inline in the cell (to keep cells compact).

**Responsive structure** (D-12): The table is wrapped in `<div class="table-scroll-wrapper">`
with `overflow-x: auto` so it scrolls horizontally on narrow screens without breaking.
Phase 4 ships the structural hooks for the full per-battery card reflow:
- `<tr>` elements receive `data-battery-id="{id}"` attributes.
- `<th>` headers receive `data-metric="{metric-key}"` attributes.
- `<td>` cells receive `data-metric="{metric-key}"` and `data-label="{th-text}"` attributes.
Phase 5's CSS will toggle these into stacked cards via `@media` and CSS display changes
without touching the HTML structure.

### 6. Saldering Disclaimer (D-14 / COMP-06)

**Trigger:** The `<button class="saldering-info-btn">` "i" in the column group header.

**Anatomy:** A `<div id="saldering-disclaimer" class="saldering-disclaimer" hidden>` placed
immediately after the table (not inside it). Toggle `hidden` attribute via JS on button
click. `aria-expanded` on the button reflects state.

**Copy (verbatim, locked — COMP-06):**
```
De kolom "met saldering" berekent de netto jaarverrekening (1:1 import versus export).
Dat is een versimpeling: vanaf 2026 is saldering al afgebouwd naar 64% van het
leveringstarief. Terugleverkosten betaal je bovendien altijd, ongeacht de saldering.
Een vloer van 50% van het kale leveringstarief blijft gelden tot en met 2030.
De kolom "zonder saldering" laat zien wat een batterij oplevert in de situatie
waarvoor je hem koopt: de toekomst na afschaffing van saldering.
```

**Styling:**
- `--color-surface` background, 1px solid `--color-border` border, border-radius 8px
- Padding: `var(--space-md)`
- `--font-size-label` (14px), `--color-text-muted`
- Max-width matches the table container width

### 7. "Rekenen..." Progress Indicator (SIM-08 / D-07)

**Trigger:** Emitted when the Comlink worker has been invoked and has not yet returned.

**Placement:** Inside `#results-region`, ABOVE the comparison table (or where the table
will appear). Replaces nothing — the stale table remains visible and dimmed during recompute.

**Anatomy:**
```html
<p class="compute-indicator" aria-live="polite" aria-busy="true">Rekenen...</p>
```

**Styling:**
- `--font-size-label` (14px), `--color-text-muted`
- No spinner animation (no CSS animation — no `animation:` property needed; plain text is sufficient)

**Stale results dimming:** While `computeIndicator` is visible, add class
`.results-stale` to the comparison table's container element. This class applies:
```css
.results-stale { opacity: 0.5; pointer-events: none; }
```
The stale table remains visible — the user sees previous results while waiting.

**Debounce:** Custom battery field input changes debounce the signal update by 400ms
before dispatching to the worker. Date-range changes and checkbox changes fire immediately
(they are discrete, not continuous input).

### 8. Empty / Loading / Error States

**No file yet (initial state):**
- `#results-region` remains visually bare (Phase 1 shell default).
- The battery picker IS shown on mount (pre-selecting Sessy 5 — D-05) so the user can
  explore specs before uploading.
- The period control and table are NOT rendered until samples are available.
- No "upload a file to begin" copy inside `#results-region` — the drop zone above already
  communicates this.

**Parsing in progress:**
- Drop zone shows "Bezig met verwerken..." (existing Phase 2 copy — unchanged).
- Battery picker and any stale results remain visible.
- `#results-region` shows nothing new until parse completes.

**Parse error:**
- Drop zone shows the structured Dutch error (existing Phase 2 behavior — unchanged).
- Battery picker remains visible.
- `#results-region` remains bare (no stale results from a failed parse).

**No battery selected:**
- Battery picker is visible, all cards unchecked (can happen if user deselects Sessy 5).
- `#results-region` shows:
  ```html
  <p class="results-empty">Selecteer minimaal één batterij om te vergelijken.</p>
  ```
  Styled: `--font-size-body`, `--color-text-muted`, padding `--space-md`.

**Computation error (worker throws):**
- `#results-region` shows:
  ```html
  <p role="alert" class="results-error">
    Berekening mislukt. Controleer of je gegevens volledig zijn en probeer het opnieuw.
  </p>
  ```
  Styled: `--font-size-body`, `--color-destructive`.
  The stale table (if any) is removed; the error message takes its place.

## Interaction Contracts

### Battery Picker

| Interaction | Behavior |
|-------------|----------|
| Click unchecked card (< 5 selected) | Check it; assign next available color slot; live recompute |
| Click checked card | Uncheck it; release color slot; live recompute |
| Click unchecked card (5 already selected) | No-op — card is `disabled`, cursor `not-allowed` |
| Click expand button on custom card | Toggle `aria-expanded`; reveal/hide form fields |
| Input in custom battery form | Debounce 400ms → validate → if valid, include in recompute |
| Tab navigation | All checkboxes and the expand button are keyboard-focusable; focus ring: `outline: 2px solid var(--color-accent)` at 2px offset |

### Period Control

| Interaction | Behavior |
|-------------|----------|
| Change "Van" date | `filterByPeriod()` on main thread → coverage indicator updates immediately → debounce 0ms (discrete) → Comlink worker call |
| Change "Tot" date | Same as above |
| "Van" > "Tot" | Clamp "Tot" to equal "Van"; show no error (valid 1-day range) |
| "Van" < merged data start | Clamp to merged data start (min attribute enforces this) |
| "Tot" > merged data end | Clamp to merged data end (max attribute enforces this) |

### Comparison Table

| Interaction | Behavior |
|-------------|----------|
| Click "i" button | Toggle saldering disclaimer visibility; `aria-expanded` updated |
| Tab through table | Standard browser table focus traversal |
| No hover interactions in Phase 4 | Tooltips / hover states are Phase 5 |

## Derived Values Contract

These computations happen in Phase 4's presentation layer (NOT in the engine):

| Metric | Formula | Notes |
|--------|---------|-------|
| kWh netto-import vermeden (zonder saldering) | `sim.totalImportKwh − sim.residualImportKwh` | = `sim.shiftedKwh` in the pure solar self-consumption model |
| kWh netto-import vermeden (met saldering) | `max(0, sim.totalImportKwh − sim.totalExportKwh) − max(0, sim.residualImportKwh − sim.residualExportKwh)` | Can be negative — display as-is (D-02) |
| Zelfverbruik % | `(sim.shiftedKwh / sim.totalImportKwh) × 100` | Clamped to [0, 100] for display only; never extrapolated |
| Marginale benutting | `sim.shiftedKwh / (battery.nominalCapacityKwh × battery.dodFraction)` | Denominator = usable capacity kWh; denominator ≥ 0.1 (guard) |
| Coverage days | `sim.periodDays` | Already computed by simulate(); display directly |

All derived values are computed in a pure helper module (node-env Vitest testable).

## Signals State Contract

Phase 4 introduces `@preact/signals-core`. Signals live in `src/state/` (new directory).

| Signal | Type | Initial Value | Who writes | Who reads |
|--------|------|---------------|------------|-----------|
| `parsedSamples` | `Signal<IntervalSample[]>` | `[]` | Drop-zone parse pipeline | Period filter, worker |
| `periodFrom` | `Signal<Date \| null>` | `null` (resolved to merged start on first parse) | Period control | Period filter |
| `periodTo` | `Signal<Date \| null>` | `null` (resolved to merged end on first parse) | Period control | Period filter |
| `selectedBatteries` | `Signal<BatteryConfig[]>` | `[BATTERY_CATALOG[0]]` (Sessy 5 pre-selected) | Battery picker | Worker |
| `customBattery` | `Signal<Partial<BatteryConfig> \| null>` | `null` | Custom battery form | Battery picker (validates + includes if valid) |
| `simResults` | `Signal<SimResult[] \| null>` | `null` | Comlink worker callback | Table renderer |
| `isComputing` | `Signal<boolean>` | `false` | Worker invocation / return | "Rekenen..." indicator |
| `computeError` | `Signal<string \| null>` | `null` | Worker error handler | Error state renderer |

`computed()` values (not stored, derived reactively):
- `filteredSamples`: `filterByPeriod(parsedSamples.value, periodFrom.value, periodTo.value)`
- `coverageDays`: derived from `filteredSamples` boundary timestamps
- `activeBatteries`: `selectedBatteries.value` with valid `customBattery` appended if present

|--------------|-------|--------------------|

for `style-src 'self'`.
