<objective>
Ship `runComparison(samples, batteries, options) → SimResult[]` in `src/domain/compare.ts` — a
thin, order-preserving `.map` over `simulate()` that returns one result per battery in input order
(SIM-06), accepting a mixed `[catalog, catalog, custom]` array (the second half of ROADMAP
criterion 6). No battery-count cap — that is a Phase 4 UI concern (BATT-05).

Purpose: This is the aggregation entry point Phase 4 wraps in a Comlink worker to drive the
comparison table. Keeping it a trivial pure fan-out makes that wrap free and keeps the order
guarantee obvious by construction.

Output: `src/domain/compare.ts`, `tests/compare.test.ts`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| batteries[] → runComparison | Each element flows into simulate(), which already range-checks a malformed custom config (T-03-03, plan 03-02). runComparison adds no new boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05 | Tampering (input) | malformed battery in the array | mitigate | Delegated to simulate()'s entry validation (T-03-03); a bad custom config throws there, so runComparison surfaces the same fail-fast error rather than producing NaN results. No additional validation needed. |
| T-03-06 | Denial of service | unbounded batteries[] length (no cap by design) | accept | Each simulate run is O(n) over a fixed sample set; Phase 4 caps the visible count at 5 in the UI and runs in a worker. A pathological large array is a Phase 4 UI concern, not a domain-layer threat. |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No installs in this plan (zero new deps). Nothing to vet. |

No network, no DOM, no I/O — a thin pure `.map` adds no meaningful attack surface beyond simulate()'s already-mitigated input validation.
</threat_model>
