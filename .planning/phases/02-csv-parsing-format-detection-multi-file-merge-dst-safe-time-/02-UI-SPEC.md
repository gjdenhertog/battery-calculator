# Phase 2 — UI Design Contract

> Visual and interaction contract for the drop-zone states and sanity readout.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Scope note:** Phase 2 is primarily a data layer. Its UI surface is deliberately minimal
> and explicitly throwaway-grade (CONTEXT.md D-01, D-03): a real drag-drop + always-visible
> file-picker button that replaces the Phase 1 `#drop-zone-region` mount point, plus a live
> sanity readout rendered after parse. No new design system, no new tokens, no inline styles
> (CSP forbids it). Phase 4 re-wires this region with signals, workers, and the comparison
> table — Phase 2's UI is a replaceable proof that the data pipeline is human-verifiable
> end-to-end.
>
> **What this spec covers:**
> 1. Drop-zone interaction states: idle / dragover / parsing / success / error
> 2. Sanity readout layout and field contract (D-08)
>
> **What this spec does not cover (explicitly deferred):**
> - Interactive period-filter date pickers (Phase 4, CONTEXT.md D-02)
> - Dutch copy/tone/tooltip pass (Phase 5, CONTEXT.md D-03)
> - Comparison table, battery selector, charts (Phases 3–5)

## Color

Inherited verbatim from Phase 1. No new color tokens. Phase 2 uses three existing tokens
for the drop-zone interaction states.

| Role | CSS Custom Property | Value | Usage in Phase 2 |
|------|--------------------|----|------------------|
| Dominant (60%) | `--color-bg` | `#ffffff` | Page background (no change) |
| Secondary (30%) | `--color-surface` | `#f4f4f5` | Drop-zone idle background, readout background |
| Border | `--color-border` | `#e4e4e7` | Drop-zone idle border, readout dividers |
| Text primary | `--color-text` | `#18181b` | Drop-zone instruction text, readout values |
| Text muted | `--color-text-muted` | `#71717a` | File-picker label, gap count, parsing status |
| Accent (10%) | `--color-accent` | `#2563eb` | Drop-zone dragover border and background tint, file-picker button border/label |
| Destructive | `--color-destructive` | `#dc2626` | Parse error messages (D-06, PRIV-03), encoding failure messages |

Accent (`#2563eb`) reserved for Phase 2: dragover state border, file-picker `<button>`
border and label text. It is NOT applied to the idle drop-zone border, readout rows, or
status messages during parsing.

Destructive (`#dc2626`) reserved for Phase 2: parse error text (file name, row, column,
expected format per DATA-09), encoding failure messages, and any hard-fail states. Soft
warnings (D-09 implausible-but-parseable values) use `--color-text-muted`, not destructive.

**Dragover background tint:** Apply `--color-accent` at 8% opacity as a background-color
on the drop-zone container during the `dragover` state. Because CSP forbids inline styles,
this must be a CSS class rule (e.g. `.drop-zone--dragover`). Use `rgb(37 99 235 / 0.08)`
as a literal value in the class rule — no new token needed for a single one-off state.

## Drop-Zone Interaction States

The `<main id="drop-zone-region">` element from Phase 1 is replaced in place. The
privacy promise `<p class="privacy-promise">` is preserved inside the region (Phase 1
locked this as a surviving DOM node). The drop-zone interactive controls are added
alongside it.

### State Machine

```
idle ──[dragenter]──► dragover ──[dragleave]──► idle
                           │
                        [drop]
                           │
                        parsing ──[error]──► error
                           │
                        [complete]
                           │
                        success
```

After `error` or `success`, the drop-zone returns to `idle` for subsequent uploads.
Multiple files are processed sequentially per the `step` callback pattern; the
parsing state lasts for the duration of all files in the drop.

### State Definitions

#### idle (default)

- **Element:** `<main id="drop-zone-region">` with class `drop-zone--idle`
- **Background:** `--color-surface` (`#f4f4f5`)
- **Border:** 1px dashed `--color-border` (`#e4e4e7`)
- **Border-radius:** 8px (inherited from global.css)
- **Min-height:** 120px (inherited from global.css)
- **Padding:** `--space-lg` (inherited from global.css)
- **Content layout:** vertical stack, centered horizontally
  - `<p class="privacy-promise">` — preserved verbatim from Phase 1
  - `<p class="drop-instruction">` — Dutch instruction text (see Copywriting)
  - `<label>` wrapping `<input type="file" multiple accept=".csv,text/csv">` — always visible (DATA-01)
- **File-picker button appearance:**
  - Rendered as a `<label>` styled as a button (with `cursor: pointer`)
  - Border: 1px solid `--color-accent`
  - Color: `--color-accent`
  - Background: `--color-bg`
  - Padding: `--space-xs` vertical, `--space-md` horizontal
  - Font-size: `--font-size-body`
  - Border-radius: 4px
  - Min-height: 44px (touch target)
  - The actual `<input type="file">` is visually hidden (clip/clip-path, not display:none,
    so it remains accessible to assistive technology)
- **ARIA:** `<main id="drop-zone-region">` carries `aria-label="Bestanden uploaden"` and
  `aria-dropeffect="copy"` (set during dragover only, removed on dragleave)

#### dragover

- Same as idle, plus class `drop-zone--dragover`
- **Border:** 1px solid `--color-accent` (`#2563eb`) — switches from dashed to solid
- **Background:** `rgb(37 99 235 / 0.08)` — 8% accent tint (class rule, not inline style)
- **Cursor:** `copy` on the drag image (controlled by `event.dataTransfer.dropEffect = 'copy'`)
- No animation — matches the "calculator utility, not a marketing surface" principle

#### parsing

- Same as idle, plus class `drop-zone--parsing`
- **Border:** 1px dashed `--color-border` (reverts from dragover)
- **Background:** `--color-surface`
- **Content added:** `<p class="parse-status">` below the file-picker label, showing
  the Dutch parsing status text (see Copywriting)
- **File-picker button:** remains visible and enabled — the user may queue additional files.
  Dropping another file while parsing is an edge case the DOM allows; the executor may
  treat this as "wait for current parse to finish then start next" — the UI does not disable
  the control (DATA-13 requires the UI to stay interactive during parsing)
- **No spinner graphic** — status text is sufficient for this throwaway-grade phase

#### success

- Same as idle, plus class `drop-zone--success`
- **Border:** 1px dashed `--color-border` (unchanged)
- **Background:** `--color-surface` (unchanged)
- **Content added:** sanity readout rendered below the drop-zone (see Sanity Readout section)
- **Parse-status text:** replaced with the success summary line (see Copywriting)
- Class `drop-zone--success` is removed and state reverts to `idle` visual appearance when a
  new file is dropped (the readout is replaced by the new parse result)

#### error

- Same as idle, plus class `drop-zone--error`
- **Border:** 1px solid `--color-destructive` (`#dc2626`)
- **Background:** `--color-surface` (unchanged)
- **Content added:** `<p class="parse-error">` rendered inside the drop-zone, below the
  file-picker label
- **Parse-error element:**
  - Color: `--color-destructive`
  - Font-size: `--font-size-body`
  - Contains: file name, row number, column, and expected format (DATA-09 contract —
    never a generic message)
- Class `drop-zone--error` and the error paragraph are removed when a new file is dropped

## Sanity Readout

Rendered below `#drop-zone-region` (not inside it) after a successful parse. The readout
element is `<section id="parse-readout" aria-label="Parseresultaten">`. It is absent from
the DOM until a successful parse completes (not hidden with `display:none` — literally not
rendered). On subsequent uploads, the readout is replaced in place.

The readout is intentionally **information-dense and unstyled beyond the token system** —
visual polish is Phase 5's job (CONTEXT.md D-03).

### Layout

Vertical stack of groups. Each group is a `<dl>` (description list) with `<dt>` for labels
and `<dd>` for values. Groups are separated by a 1px `--color-border` divider.

```
<section id="parse-readout">
  <div class="readout-group">          ← Summary group (DATA-11 fields)
    <dl>...</dl>
  </div>
  <div class="readout-divider"></div>
  <div class="readout-group">          ← Per-file group (D-08 fields, one per file)
    <dl>...</dl>
  </div>
  <div class="readout-divider"></div>
  <div class="readout-group">          ← Warnings group (D-09 soft warnings, if any)
    <dl>...</dl>
  </div>
</section>
```

The warnings group is omitted entirely if there are no soft warnings.

### Readout Styling

- **Background:** `--color-surface`
- **Border:** 1px solid `--color-border`
- **Border-radius:** 8px
- **Padding:** `--space-lg`
- **Margin-top:** `--space-xl` (gap between drop-zone and readout)
- **`<dt>` elements:** font-size `--font-size-label` (14px), weight 400, color `--color-text-muted`
- **`<dd>` elements:** font-size `--font-size-body` (16px), weight 400, color `--color-text`, margin-left 0
- **Divider (`.readout-divider`):** 1px solid `--color-border`, margin `--space-md` vertical

### Readout Field Contract

The following fields are required in the readout. Labels are functional Dutch (D-03 — not
the final tone-polished copy, which is Phase 5). Values are formatted as described.

#### Summary Group (DATA-11 — mandatory)

| Field | Dutch label | Value format | Notes |
|-------|-------------|--------------|-------|
| File count | `Bestanden` | Integer (e.g. `2`) | Count of successfully parsed files |
| Total rows | `Rijen` | Integer with thousands separator (e.g. `52.438`) | Sum across all files before dedup |
| Date range | `Periode` | `DD-MM-YYYY – DD-MM-YYYY` (NL date format) | Earliest and latest timestamp in merged series |
| Total import | `Totaal netafname` | `{value} kWh` at 1 decimal place | Sum of `gridImportKwh` across merged series |
| Total export | `Totaal teruglevering` | `{value} kWh` at 1 decimal place | Sum of `gridExportKwh` across merged series |
| Gap count | `Ontbrekende intervallen` | Integer, or `Geen` if 0 | Count of expected intervals with no data (D-04) |

#### Per-File Group (D-08 — mandatory, repeated per uploaded file)

| Field | Dutch label | Value format | Notes |
|-------|-------------|--------------|-------|
| File name | `Bestand` | Filename string | As provided by the browser's File object |
| Detected resolution | `Resolutie` | `15 minuten` or `Uur` | Dominant cadence inferred from data |
| Series type | `Type meting` | `Cumulatief` or `Interval` | `series_type` declared by adapter |
| Monotonicity check | `Monotoon` | `Ja` or `Nee — mogelijke meterswap op rij {n}` | D-08: monotonicity check result; row number on failure |
| First-interval flag | `Eerste interval` | `OK` or `Aanpassing toegepast` | D-08: first-interval anomaly per DATA-05 |
| Rows contributed | `Rijen meegenomen` | Integer | Samples from this file that appear in the merged series |
| Rows overridden | `Rijen overschreven` | Integer | Samples overridden by a finer-resolution file (DATA-10 visibility per D-08) |
| Encoding used | `Encoding` | `UTF-8` or `Windows-1252 (fallback)` | D-07: encoding actually used |

#### Warnings Group (D-09 — optional, only if soft warnings exist)

| Field | Dutch label | Value format | Notes |
|-------|-------------|--------------|-------|
| Warning entries | `Aandachtspunten` | One `<dd>` per warning | Implausible but parseable values; never blocks |

Each warning entry is a single line: functional Dutch description of what looked unusual.
Example format (not final copy): `Bestand "export.csv", rij 4231: ongebruikelijk hoog importwaarde (8,4 kWh).`
Color: `--color-text-muted` (not destructive — these are soft warnings, not errors).

## Copywriting Contract

All copy is in Dutch. Labels are functional and unpolished (D-03) — the tone/terminology
audit is Phase 5's job. The Phase 5 pass will replace these strings without structural
changes to the DOM.

| Element | Dutch copy | Notes |
|---------|-----------|-------|
| Drop instruction | `Sleep een of meer CSV-bestanden hierheen` | Inside drop-zone, idle state |
| File-picker button label | `Of kies bestanden` | `<label>` styled as button, always visible (DATA-01) |
| Parsing status | `Bezig met verwerken...` | Shown during `parsing` state |
| Success summary | `{n} bestand(en) verwerkt.` | Shown in drop-zone after parse (replaced by readout below) |
| Empty readout | Not applicable | Readout is absent from DOM until success; no empty-state needed |
| Error — parse fail | `Fout in bestand "{filename}", rij {n}, kolom "{col}": {expected}.` | DATA-09 contract — exact fields required; `--color-destructive` |
| Error — encoding fail | `Bestand "{filename}" heeft een niet-ondersteunde encoding.` | Only shown if UTF-8 and Windows-1252 both fail |
| Soft warning prefix | `Aandachtspunt:` | D-09 prefix for implausible-but-parseable warnings |
| Privacy promise | `Je data blijft op je eigen apparaat — open je netwerktabblad en je ziet 0 verzoeken na het laden` | Verbatim locked from Phase 1 PRIV-02 — not changed in Phase 2 |

**Primary CTA:** `Of kies bestanden` — this is the only explicit action label in Phase 2.
It is rendered on an always-visible file-picker `<label>` (DATA-01 requires the button
always be visible, not just a drag target).

**Terminology rules (inherited from Phase 1 UI-SPEC, UX-05):**
- Never write "zonne-opwekking", "solar production", or "solar generation"
- Always write "teruglevering" or "stroom teruggeleverd aan het net" for grid export
- `gridExportKwh` is labeled `Totaal teruglevering` in the readout (not "solar export" or
  "opwekking")

**No destructive actions in Phase 2.** No confirmation dialogs are needed. Dropping a
new file implicitly replaces the current readout — no confirmation is required for this
implicit replacement (it is not destructive).

## Accessibility Contract

| Element | Requirement |
|---------|-------------|
| `<main id="drop-zone-region">` | `aria-label="Bestanden uploaden"` |
| Drop-zone during dragover | `aria-dropeffect="copy"` on the element (set/removed by JS) |
| File `<input>` | Visually hidden but accessible: `clip-path: inset(50%)`, not `display:none` |
| File `<label>` | Associated with `<input>` via `for`/`id` |
| Error paragraph | `role="alert"` so screen readers announce parse errors without focus management |
| Parse-status paragraph | `aria-live="polite"` so readers announce parsing completion |
| `<section id="parse-readout">` | `aria-label="Parseresultaten"` |
| Description lists | Semantic `<dl>/<dt>/<dd>` — no tables for readout fields |
| Touch targets | File-picker label: `min-height: 44px` (WCAG 2.5.5) |

## CSP Compatibility Contract

Inherited from Phase 1 UI-SPEC. Phase 2 introduces no new CSP requirements.

| CSP directive | Value | Phase 2 implication |
|--------------|-------|---------------------|
| `style-src` | `'self'` | All drop-zone states controlled via CSS classes; no `style=` attributes |
| `connect-src` | `'none'` | PapaParse `worker: true` uses a same-origin blob worker — confirm in planning that the blob URL does not violate CSP; if needed, add `worker-src blob:` to the build-only CSP |
| `script-src` | `'self'` | PapaParse and date-fns bundled by Vite; no dynamic `import()` from external URLs |
| `img-src` | `'self'` | No images in Phase 2 UI |
| `font-src` | `'self'` | System fonts only (inherited) |

**PapaParse worker CSP note:** PapaParse `worker: true` spawns a blob worker internally.
The Phase 1 maximal-lockdown CSP uses `worker-src 'none'` implicitly (inherited from
`default-src 'none'`). Planning must resolve this: either add `worker-src blob:` to the
build-only CSP plugin, or configure PapaParse to use `worker: false` with an explicit
`?worker` Vite import instead. This is a planning decision, not a design decision — but
the UI-SPEC flags it so the planner does not miss it.
